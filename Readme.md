## Конфигурация приложения (`config.py`)

### Назначение
Файл `config.py` — это центральный узел управления всеми настройками бота. Он отвечает за:
- Чтение переменных окружения из файла `.env`
- Валидацию и преобразование типов (строка → число, строка → булево значение и т.д.)
- Предоставление единого объекта `settings`, к которому можно обратиться из любого модуля проекта

### Почему это важно
- **Безопасность** – токены, пароли и ключи API не хранятся в коде, а лежат в `.env`, который не попадает в Git.
- **Гибкость** – можно менять поведение бота (интервал поиска, таймауты, URL API) без изменения кода, просто отредактировав `.env`.
- **Удобство** – все настройки собраны в одном месте, с автодополнением и проверкой типов благодаря Pydantic.

### Как это работает
Мы используем библиотеку `pydantic-settings`. Класс `Settings` наследуется от `BaseSettings` и автоматически:
1. Ищет файл `.env` в корне проекта.
2. Сопоставляет переменные окружения с полями класса.
3. Преобразует значения в нужные типы (например, `"10"` → `10`).
4. Применяет валидаторы (например, проверяет формат токена).

### Использование в коде
```python
from config import settings

# Получить интервал поиска
interval = settings.search_interval_minutes

# Создать клиент API с таймаутом
client = TravelataAPI(settings.travelata_api_url, timeout=settings.api_request_timeout)

# Проверить, включён ли дебаг-режим
if settings.debug_mode:
    print("Запущен в отладочном режиме")
```

## Модели данных (models.py)

В этом модуле описаны все сущности, с которыми работает бот. Модели разделены на три категории:

### 1. Модели для API Travelata
Описывают структуру ответов от внешнего API: страны, курорты, отели, категории отелей, типы питания, города вылета. Используются для валидации и преобразования данных.

### 2. Модель тура (Tour)
Основная сущность, получаемая из эндпоинта `cheapestTours`. Содержит всю информацию о предложении: цена, отель, даты, ссылки. Используется для отображения пользователю и для отслеживания изменений цен.

### 3. Внутренние модели для базы данных
- **User** – данные пользователя Telegram.
- **SearchCriteria** – сохранённые критерии поиска (подписки на определённые условия).
- **Subscription** – связь пользователя с критериями и статус подписки.
- **PriceHistory** – история цен на конкретный тур (нужна для уведомлений о снижении цены).

Все внутренние модели используют Pydantic для валидации и будут преобразованы в SQLAlchemy-модели на следующем этапе.


## Клиент для API Travelata (travelata_api.py)

Модуль отвечает за взаимодействие с внешним API Travelata. Реализует:
- HTTP Basic Auth (логин/пароль)
- Асинхронные запросы через aiohttp
- Преобразование JSON-ответов в Pydantic-модели
- Обработку ошибок и повторные попытки

Все методы возвращают типизированные списки объектов (Country, Resort, Tour и т.д.).


## Объяснение ключевых конструкций кода

### logger = logging.getLogger(__name__)
Что это?
Это стандартный механизм логирования в Python.
__name__ — это имя текущего модуля (например, travelata_api).
getLogger(__name__) создаёт или получает логгер с именем модуля. Это позволяет в логах видеть, из какого файла пришло сообщение.
Зачем?
Чтобы в процессе работы бота выводить отладочные сообщения, предупреждения и ошибки. Это помогает отслеживать, что происходит внутри клиента: какие запросы отправляются, какие ошибки возникают. В будущем вы сможете настроить уровень логирования (INFO, DEBUG, ERROR) через конфигурацию.

### self.retry_count
Что это?
Это атрибут экземпляра класса (поле объекта). В __init__ вы присваиваете значение self.retry_count = retry_count or settings.api_retry_count.
Зачем?
Чтобы хранить количество попыток повторного запроса при ошибках. Используется в методе _request(): for attempt in range(1, retries + 1). Если запрос не удался, цикл повторяется self.retry_count раз.
Почему self?
self — ссылка на текущий объект. Без self это была бы локальная переменная, доступная только внутри метода __init__. С self мы сохраняем значение во всём объекте, чтобы другие методы могли его использовать.

### async with self._session.get(url, params=params, auth=auth) as resp:
Что это?
self._session — объект aiohttp.ClientSession, который управляет пулом соединений.
get() — отправляет GET-запрос по указанному URL с параметрами и авторизацией.
auth=auth — передаёт объект aiohttp.BasicAuth, который добавляет заголовок Authorization: Basic ....
async with — асинхронный контекстный менеджер. Он автоматически закрывает ответ (освобождает ресурсы) после выхода из блока, даже если произошла ошибка.
Зачем?
Это стандартный паттерн для асинхронных HTTP-запросов в Python. Он обеспечивает безопасную работу с сетевыми соединениями.

### resp.raise_for_status()
Что это?
Метод объекта ответа (ClientResponse). Проверяет статус-код: если он не в диапазоне 200–299, выбрасывает исключение ClientResponseError с информацией о коде ошибки.
Зачем?
Чтобы не писать вручную проверку каждого статуса. В коде вы уже обрабатываете 429 отдельно, а для всех остальных (4xx, 5xx) raise_for_status() автоматически выбросит исключение, которое вы ловите в except ClientResponseError.

### data = await resp.json()
Что это?
Асинхронный метод, который читает тело ответа и парсит его из JSON в словарь Python.
Зачем?
API возвращает данные в формате JSON. Чтобы работать с ними как с объектами Python, нужно распарсить строку.

### data = await self._request("partners/directory/countries")
Здесь вы вызываете свой же метод _request(), который выполняет запрос и возвращает уже распарсенный JSON (содержимое поля result). Это делает код чище: не нужно каждый раз писать логику запроса.

### return [Resort(**item) for item in data]
Это генератор списка (list comprehension).
data — это список словарей, полученных от API.
Resort(**item) — создаёт объект модели Resort, передавая словарь item как именованные аргументы. Оператор ** распаковывает словарь: Resort(id=..., name=..., ...).
В результате возвращается список объектов Pydantic, с которыми удобно работать (есть валидация типов, автодополнение в IDE).
Зачем?
Преобразование сырых JSON-данных в строго типизированные объекты упрощает дальнейшую работу и снижает вероятность ошибок.

## Модуль `search_engine.py` – гибкий поиск и ранжирование

### Назначение
Модуль реализует логику оценки соответствия каждого тура критериям пользователя и сортировку результатов по проценту совпадения. Это позволяет боту предлагать туры, даже если они не идеально подходят под все параметры.

### Что нужно доработать в будущем
Справочник соответствия resort → country: чтобы проверять страну, нужно иметь кэш или вызывать get_resorts() и get_countries().
Частичное совпадение для дат и ночей: можно вычислять расстояние до интервала.
Учёт предпочтений: хранить историю согласий и корректировать веса.
Оптимизация: кэшировать справочники, чтобы не делать лишних запросов.
